## 2. Historia zmian studentów
System został rozbudowany o mechanizm historii zmian studentów. Oznacza to, że operacje modyfikujące stan danych nie kończą się już wyłącznie aktualizacją rekordu w tabeli głównej, lecz pozostawiają również trwały zapis w tabeli historii.

W praktyce zapisujemy informację o tym: 
* jaka operacja została wykonana
* którego studenta dotyczyła
* jaki był stan obiektu przed zmianą
* jaki jest stan po zmianie
* kiedy dokładnie dana operacja została zarejestrowana 

System nie przechowuje jedynie aktualnej wersji danych, lecz również ich kontekst historyczny.

Rozszerzenie to obejmuje rejestrację operacji POST, PATCH, PUT, DELETE dla każdego studenta. W rezultacie możliwe staje się prześledzenie całego cyklu życia rekordu w systemie.

Tego typu mechanizm jest szczególnie przydatny w sytuacjach, w których chcemy analizować przebieg zmian danych, odtwarzać wcześniejsze stany obiektu lub diagnozować błędy użytkowników.

### Nowa tabela historii

Aby możliwe było trwałe przechowywanie historii operacji wykonywanych na studentach, musimy rozbudować skrypt `001_init.sql` o nową tabelę `student_history`.

Dotychczas baza danych przechowywała wyłącznie aktualny stan rekordu studenta. Oznacza to, że po wykonaniu operacji `PUT`, `PATCH` lub `DELETE` poprzednia wersja danych była tracona. Wprowadzenie dodatkowej tabeli pozwala nam zachować informację o przebiegu zmian w czasie.


```sql
CREATE TABLE IF NOT EXISTS student_history (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    student_id INTEGER NOT NULL,
    action VARCHAR NOT NULL,
    previous_state JSONB NOT NULL,
    current_state JSONB NOT NULL,
    changed_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_student_history_student_id
    ON student_history(student_id);
```
Każdy rekord w tej tabeli opisuje jedno zdarzenie wykonane na studencie.
Przechowujemy tutaj:
* id – identyfikator wpisu historii
* student_id – identyfikator studenta, którego dotyczy operacja
* action – nazwę wykonanej operacji
* previous_state – stan obiektu przed zmianą
* current_state – stan obiektu po zmianie
* changed_at – moment zapisania zdarzenia

Dlaczego `student_id` nie jest kluczem obcym?<br>
W klasycznym modelu relacyjnym naturalnym pomysłem mogłoby być dodanie relacji `FOREIGN KEY` z tabelą **students** aby zmiana w historii była bezpośrednio z nim powiązana. 
W przypadku historii zmian może nie być to jednak najlepszym rozwiązaniem.

Gdyby `student_id` był kluczem obcym wskazującym na tabelę **students**, to po usunięciu rekordu studenta, tabela z rekordami historii musiałaby odpowiednio zareagować na zerwanie relacji. W zależności od konfiguracji mogłoby to prowadzić do:
* usunięcia wpisów historii
* ustawienia wartości `student_id` na `NULL`
* błędu integralności bazy

Żadne z tych zachowań nie jest dla nas korzystne. W naszej historii zmian chcemy zachować informację, że student o określonym identyfikatorze istniał i wykonano na nim daną operację, nawet jeśli rekord główny został już fizycznie usunięty z tabeli **students**. Takie rozwiązanie jest lepsze dla tabeli historii, ponieważ rekord historyczny ma być trwałym zapisem zdarzenia z przeszłości, a nie odzwierciedleniem aktualnego stanu relacji w bazie danych.

Dlaczego używamy pól `previous_state` i `current_state`?
W historii zmian nie chcemy zapisywać jedynie informacji, że „coś zostało zmienione”. Chcemy również wiedzieć co dokładnie uległo zmianie.
Dlatego przechowujemy dwa stany obiektu:
* stan przed operacją
* stan po operacji

Przechowywanie stanu obiektu sprzed operacji pozwala również sprawdzić, czy pomiędzy kolejnymi zmianami rekord nie został zmieniony w sposób nieautoryzowany, z pominięciem mechanizmów udostępnionych przez aplikację.

### 2.1. Model ORM dla historii zmian studentów

Po przygotowaniu nowej tabeli w bazie danych kolejnym krokiem jest utworzenie odpowiadającego jej modelu ORM `student_history_orm.py`. 
```python
class StudentHistoryORM(Base):
    __tablename__ = "student_history"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    student_id: Mapped[int] = mapped_column(
        Integer,
        nullable=False,
        index=True  
    )
    action: Mapped[str] = mapped_column(String, nullable=False)
    previous_state: Mapped[dict] = mapped_column(JSON, nullable=False)
    current_state: Mapped[dict] = mapped_column(JSON, nullable=False)
    changed_at: Mapped[datetime] = mapped_column(
        DateTime,
        nullable=False,
        server_default=func.now()
    )
```
### 2.2. Repozytorium historii zmian studentów

Po zdefiniowaniu modelu ORM kolejnym krokiem jest przygotowanie warstwy dostępu do danych odpowiedzialnej za obsługę historii zmian. W naszym projekcie rolę tę pełni plik `student_history_repository.py`.

Pierwsza funkcja repozytorium odpowiada za pobranie wszystkich wpisów historii dla wskazanego identyfikatora studenta:
```python
def get_student_history_by_student_id(db: Session, student_id: int):
    query = (
        select(StudentHistoryORM)
        .where(StudentHistoryORM.student_id == student_id)
        .order_by(
            desc(StudentHistoryORM.changed_at), 
            desc(StudentHistoryORM.id)
        ) 
    )
    #sortujemy względem czasu zmiany, w przypadku gdyby dwa wpisy otrzymały bardzo zbliżony znacznik czasu,
    #id rekordu pozwoli zachować odpowiednią kolejność wyników
    result = db.execute(query)
    return result.scalars().all()
```

Druga funkcja repozytorium odpowiada za zapisanie nowego wpisu historii do bazy danych:
```python
def add_student_history(db: Session, history_entry: StudentHistoryORM):
    db.add(history_entry)
    db.commit()
    db.refresh(history_entry)
    return history_entry
```

### 2.4. Najważniejsze zmiany w warstwie serwisowej

Najistotniejsze rozszerzenie całego mechanizmu historii zmian zostało wprowadzone w warstwie serwisowej `student_service.py`. To właśnie tutaj aplikacja podejmuje decyzję, **kiedy** należy zapisać historię, **jaką operację** należy zarejestrować oraz **jaki stan obiektu** powinien zostać zapisany przed i po wykonaniu zmiany.

#### Wydzielenie funkcji budującej snapshot obiektu
Jedną z najważniejszych zmian było dodanie funkcji `_build_student_snapshot`, której zadaniem jest utworzenie pełnego obrazu aktualnego stanu studenta.

```python
def _build_student_snapshot(student: StudentORM) -> dict:
    field = student.field_of_study
    faculty = field.faculty

    return {
        "id": student.id,
        "name": student.name,
        "lastname": student.lastname,
        "student_code": student.student_code,
        "email": student.email,
        "ects_points": student.ects_points,
        "field_of_study": {
            "id": field.id,
            "name": field.name,
            "min_ects": field.min_ects,
            "max_ects": field.max_ects,
            "faculty": {
                "id": faculty.id,
                "name": faculty.name,
            },
        },
    }
```
Funkcja ta przygotowuje słownik zawierający pełny stan obiektu studenta wraz z informacją o kierunku studiów oraz wydziale. Dzięki temu historia nie ogranicza się do prostego komunikatu o wykonanej operacji, lecz przechowuje rzeczywistą zawartość obiektu w danym momencie.

#### Wydzielenie funkcji pomocniczej zapisującej  historię
Funkcja `_save_student_history` centralizuje logikę tworzenia wpisu historycznego. Zamiast powielać kod odpowiedzialny za tworzenie obiektu StudentHistoryORM w każdej funkcji, logika została zamknięta w jednym helperze. 

```python
def _save_student_history(
    db: Session,
    student_id: int | None,
    previous_state: dict,
    current_state: dict,
    action: str,
):
    history_entry = StudentHistoryORM(
        student_id=student_id,
        action=action,
        previous_state=previous_state,
        current_state=current_state,
    )
    return add_student_history(db, history_entry)
```
Funkcja buduje obiekt historii dla każdej operacji i przekazuje go do repozytorium, które wykonuje właściwy zapis w bazie danych.

#### Dodanie odczytu historii studenta
Funkcja `list_student_history` jest odpowiedzialna za pobranie historii zmian konkretnego studenta.
```python
def list_student_history(db: Session, student_id: int):
    return get_student_history_by_student_id(db, student_id)
```

#### 2.5. Rejestracja historii zmian

W funkcjach modyfikujących stan studenta dodany został fragment odpowiedzialny za zapisanie wpisu do historii zmian. Mechanizm ten działa analogicznie dla operacji `POST`, `PUT`, `PATCH` oraz `DELETE`, przy czym każdej z nich przypisano odpowiednią wartość pola `action`:

- `POST` → `CREATE`
- `PUT` → `REPLACE`
- `PATCH` → `UPDATE`
- `DELETE` → `DELETE`

Najbardziej reprezentatywnym przykładem jest operacja częściowej aktualizacji, czyli `PATCH`. W tym przypadku przed zapisaniem zmian tworzony jest snapshot aktualnego stanu obiektu, następnie wykonywana jest właściwa aktualizacja rekordu, po czym tworzony jest snapshot stanu po zmianie. Oba te stany zostają następnie zapisane w historii wraz z nazwą wykonanej operacji.

```python
def patch_student(db: Session, student_id: int, payload: StudentUpdate):
    student = get_student_by_id(db, student_id)
    if student is None:
        return None

    previous_state = _build_student_snapshot(student) # zapis stanu obiektu przed modyfikacją

    #logika walidacyjna

    updated_student = save_student(db, student)

    current_state = _build_student_snapshot(updated_student)  # zapis stanu obiektu po modyfikacji

    _save_student_history(
        db=db,
        student_id=updated_student.id,
        previous_state=previous_state,
        current_state=current_state,
        action="UPDATE",
    ) # utworzenie wpisu historii dla operacji UPDATE

    return updated_student
```
Taki sam schemat stosowany jest również dla pozostałych operacji modyfikujących. Różnica polega jedynie na nazwie akcji oraz zawartości stanów granicznych. W przypadku `POST` pole `previous_state` pozostaje puste, ponieważ obiekt nie istniał jeszcze przed utworzeniem. Z kolei przy `DELETE` puste pozostaje `current_state`, ponieważ po wykonaniu operacji rekord nie istnieje już w aktywnej tabeli studentów.

## 3. Testy
Testowanie oprogramowania jest jednym z podstawowych elementów profesjonalnego procesu wytwarzania aplikacji. Jego celem jest sprawdzenie, czy system działa zgodnie z założeniami funkcjonalnymi.

W praktyce testy polegają na automatycznym uruchamianiu fragmentów aplikacji z określonymi danymi wejściowymi oraz sprawdzaniu, czy otrzymany wynik jest zgodny z oczekiwanym rezultatem.

Najczęściej testy przyjmują formę kodu, który:
1. przygotowuje środowisko testowe
2. wywołuje testowaną funkcję lub endpoint
3. sprawdza wynik działania aplikacji

```python
#Przykładowy test:
def test_addition():
    result = 2 + 2
    assert result == 4
```
Instrukcja assert jest kluczowym elementem testów, ponieważ oznacza ona, że dany warunek musi być spełniony. Jeżeli warunek nie jest spełniony, test kończy się niepowodzeniem.

### 3.1. Rodzaje testów
W inżynierii oprogramowania wyróżnia się kilka podstawowych rodzajów testów, które różnią się poziomem szczegółowości oraz zakresem sprawdzanej funkcjonalności.
#### `Testy jednostkowe (Unit tests)`
Testy jednostkowe sprawdzają pojedyncze elementy systemu, najczęściej pojedyncze funkcje lub klasy.
Ich celem jest weryfikacja logiki w izolacji od reszty systemu.

```python
#Przykładowe testy:
def calculate_ects_sum(a, b):
    return a + b

def test_calculate_ects_sum():
    assert calculate_ects_sum(20, 25) == 45
```
Test jednostkowy sprawdza wyłącznie poprawność działania **jednej** funkcji.

#### `Testy integracyjne (Integration tests)`
Testy integracyjne sprawdzają współpracę wielu elementów systemu jednocześnie.
W przypadku aplikacji REST API oznacza to najczęściej testowanie:

* endpointów HTTP
* logiki serwisowej
* komunikacji z bazą danych
* walidacji danych
```python
#Przykładowy test:
response = client.get("/students")

assert response.status_code == 200
```
Testy integracyjne są szczególnie istotne w aplikacjach webowych, ponieważ pozwalają sprawdzić rzeczywiste działanie systemu z punktu widzenia klienta API.

### 3.2. Framework testowy **pytest**

Jest to nowoczesny framework testowym, który umożliwia tworzenie testów w bardzo prosty i czytelny sposób. Framework ten jest szeroko wykorzystywany zarówno w projektach open-source, jak i w profesjonalnych systemach produkcyjnych.

#### `fixture db_session`

Poniższy element przygotowuje środowisko bazy danych dla pojedynczego testu.
Każdy test działa w osobnej transakcji, która po jego zakończeniu zostaje cofnięta.
Dzięki temu testy nie zostawiają danych w bazie i nie wpływają na siebie nawzajem.

```python
@pytest.fixture(scope="function")
def db_session():
    """
    Przygotowuje sesję bazy danych dla testu.
    Każdy test działa w jednej transakcji, która na końcu zostaje cofnięta.
    Dzięki temu testy nie zostawiają danych w bazie.
    """
    #otwarcie połączenia z bazą danych i rozpoczęcie głównej transakcji
    connection = engine.connect()
    outer_transaction = connection.begin()

    #sesja SQLAlchemy powiązana z tym połączeniem
    session = SessionLocal(bind=connection)
    nested_transaction = connection.begin_nested()

    #event listener, który automatycznie tworzy nowy savepoint
    @event.listens_for(session, "after_transaction_end")
    def restart_savepoint(sess, transaction):
        nonlocal nested_transaction

        #utworzenie zagnieżdżonej transakcji (savepointu)
        if not nested_transaction.is_active:
            nested_transaction = connection.begin_nested()

    try:
        #Przekazanie utworzonej sesji do testu
        yield session

    finally:
        session.close()

        #Cofnięcie zapisu do bazy danych 
        outer_transaction.rollback()
        connection.close()
```
Kroki działającego programu:

1. Na początku otwierane jest połączenie z bazą danych oraz rozpoczynana jest główna transakcja dla testu. 
Transakcję można traktować jako zestaw zmian w bazie danych, który można później zatwierdzić (commit) lub cofnąć (rollback).
W testach używamy rollback, aby zmiany wykonane podczas testu nie zostały zapisane na stałe.
2. Następnie tworzona jest sesja SQLAlchemy powiązana z tym połączeniem.
Dzięki temu przygotowana zostaje sesja testowa powiązana z kontrolowaną transakcją. Sesja ta może następnie zostać wykorzystana przez endpointy API po nadpisaniu zależności `get_db`.
Utworzony Savepoint pozwala utrzymać kontrolę nad transakcją testową nawet wtedy, gdy kod aplikacji wykonuje `commit()` podczas zapisu do bazy.
3. Następnie definiowany jest event listener, który automatycznie tworzy nowy savepoint, jeśli poprzedni zostanie zakończony.Dzięki temu test może wykonywać wiele operacji zapisu w bazie, a całość nadal pozostaje w jednej transakcji testowej.
4. Uruchomienie instrukcji `yield session`, która przekazuje przygotowaną sesję do testu.
Od tego momentu zarówno test, jak i endpointy aplikacji korzystają z tej samej sesji bazy danych.
5. Wykonanie bloku `finally`, po wykonaniu testu. Najważniejszą operacją jest `rollback`, który cofa wszystkie zmiany wykonane podczas testu.
Dzięki temu baza danych wraca do stanu sprzed uruchomienia testu, a kolejne testy rozpoczynają pracę na czystym stanie bazy.

#### `fixture client`
Poniższy element tworzy klienta testowego aplikacji oraz nadpisuje zależność `get_db`, aby endpointy FastAPI korzystały z sesji przygotowanej wcześniej w `db_session`.

```python
@pytest.fixture(scope="function")
def client(db_session):
    """
    Nadpisuje zależność get_db tak, aby endpointy FastAPI używały sesji testowej.
    Dzięki temu również wewnętrzne commit() wykonywane przez repozytorium
    pozostają zamknięte w transakcji testowej i zostaną cofnięte po teście.
    """
    #zastąpienie standardowej zależność `get_db` używanej w aplikacji.
    def override_get_db():
        try:
            yield db_session
        finally:
            pass

    #nadpisanie zależności w aplikacji
    app.dependency_overrides[get_db] = override_get_db

    try:
        #utworzenie klienta testowego FastAPI
        yield TestClient(app)
    finally:
        #usuwanie nadpisanych zależności
        app.dependency_overrides.clear()
```
Kroki w działającym programie:
1. Na początku fixture definiowana jest funkcja `override_get_db`, która zastępuje standardową zależność `get_db` używaną w aplikacji.Normalnie endpointy FastAPI pobierają sesję bazy danych przez zależność `get_db`.
W testach chcemy jednak, aby korzystały z sesji utworzonej w fixture `db_session`, która działa w kontrolowanej transakcji testowej.
2. Nadpisanie zależności w aplikacji. FastAPI umożliwia podmianę zależności używanych przez endpointy.
Dzięki temu podczas testów zamiast standardowego `get_db()` używana jest funkcja `override_get_db`, która zwraca sesję testową.
3. Utworzenie klienta testowego FastAPI.
4. Po zakończeniu testu wykonywany jest blok finally.Usuwane są wszystkie nadpisane zależności aplikacji.
Dzięki temu konfiguracja testowa nie wpływa na kolejne testy ani na działanie aplikacji poza środowiskiem testowym.
