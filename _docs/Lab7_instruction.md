## Model użytkownika i sesji
Podstawą mechanizmu uwierzytelniania jest wprowadzenie dwóch nowych bytów w bazie danych:
* operator systemu (`operators`)
* sesja operatora (`operator_sessions`)

Operator reprezentuje użytkownika systemu, który wykonuje operacje na danych (studentach). Sesja natomiast przechowuje informację o zalogowanym użytkowniku i umożliwia identyfikację użytkownika w kolejnych żądaniach HTTP.

```sql
--app/db/init/003_identity.sql
CREATE TABLE IF NOT EXISTS operators (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email VARCHAR NOT NULL UNIQUE,
    password_hash VARCHAR NOT NULL,
    first_name VARCHAR NOT NULL,
    last_name VARCHAR NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE TABLE IF NOT EXISTS operator_sessions (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    operator_id INTEGER NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
    session_token VARCHAR NOT NULL UNIQUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_used_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## Hashowanie haseł 
Hasło użytkownika jest jedną z najbardziej wrażliwych informacji w systemie. Z tego powodu absolutnie nie może być przechowywane w bazie danych w postaci jawnej typu `password = "StrongPass1!"`. Gdyby takie dane wyciekły przez błąd albo atak, wszystkie konta użytkowników byłyby natychmiast narażone. Co więcej, użytkownicy często używają tych samych haseł w różnych serwisach, więc konsekwencje takiego wycieku są znacznie poważniejsze.

Zamiast tego stosuje się mechanizm **hashowania haseł**. Hashowanie polega na przekształceniu hasła do postaci skrótu (hasha), czyli ciągu znaków, którego nie da się w praktyce odwrócić, ale można go ponownie obliczyć dla danego wejścia.

Czyli zamiast przechowywać hasło, przechowujemy ciąg liter, liczb lub znaków `a3f91c0d...e21f`. 

Podczas logowania użytkownik podaje hasło, system oblicza jego hash i porównuje go z tym zapisanym w bazie. Jeśli są one identyczne następuje logowanie. 

Zwykły hash typu SHA256 nie jest bezpieczny, ponieważ istnieją 
* **rainbow tables** - gotowe bazy hashy dla popularnych haseł
* ataki brute-force - ataki sprawdzające miliony kombinacji w krótkim czasie
Nowoczesne sprzęty (GPU) są w stanie liczyć miliony hashy na sekundę, dlatego prosty hash typu SHA256 nie jest wystarczająco bezpieczny.

Zamiast tego używa się tzw KDF - Key Derivation Function. Za ich pomocą możliwe jest utrudnienie ataków brute-force i znaczącego zwiększenia kosztu łamania hasła.
Do najpopularniejszych należą:
* bcrypt
* Argon2
* PBKDF2

W tym projekcie wykorzystywany jest algorytm PBKDF2, ze względu na fakt, że jest on dostępny w standardowej bibliotece Pythona hashlib, nie wymaga instalacji dodatkowych pakietów i jest stosowany w wielu systemach produkcyjnych.

Jego implementacja wygląda następująco:
```python
# app/identity/service/password_hasher.py
PBKDF2_ITERATIONS = 100_000
SALT_BYTES = 16

def hash_password(password: str) -> str:
    salt = secrets.token_hex(SALT_BYTES)
    derived_key = hashlib.pbkdf2_hmac(
        "sha256",
        password.encode("utf-8"),
        salt.encode("utf-8"),
        PBKDF2_ITERATIONS,
    )
    return f"{salt}${derived_key.hex()}"
```

Na początku definiujemy dwie stałe `PBKDF2_ITERATIONS` i `SALT_BYTES `. Pierwsza z nich określa, ile razy wykonywana jest operacja hashowania. Im większa ta liczba, tym wolniejsze obliczenie hasha, ale jednocześnie trudniejsze złamanie hasła przez atak brute-force. `SALT_BYTES ` określa długość losowego salta.

Salt to losowa wartość dodawana do hasła przed jego hashowaniem.<br>
Bez salta:<br>
```
password = "123456"
hash = SHA256(password)
Jeśli 100 użytkowników ma to samo hasło to wszystkie hashe będą identyczne. Oznacza to, że łatwo zauważyć, że użytkownicy używają tych samych haseł, a złamanie jednego hasła może umożliwić dostęp do wielu kont.
```

Z saltem:
```
password = "123456"<br>
salt = "a1b2"<br>
hash = SHA256("123456a1b2")

password = "123456"<br>
salt = "f9k3"<br>
hash = SHA256("123456f9k3")<br>
```

Mimo że hasło jest takie samo, wynik jest inny, ponieważ użyty został inny salt.

W naszym kodzie salt generowany jest losowo `salt = secrets.token_hex(SALT_BYTES)`.
Na końcu zapisujemy w bazie zarówno salt, jak i hash `return f"{salt}${derived_key.hex()}"`.

Podczas logowania nie porównujemy haseł bezpośrednio, tylko ponownie liczymy hash i porównujemy wynik:

```python
# app/identity/service/password_hasher.py
def verify_password(password: str, stored_hash: str) -> bool:
    salt, expected = stored_hash.split("$", 1)

    derived = hashlib.pbkdf2_hmac(
        "sha256",
        password.encode(),
        salt.encode(),
        PBKDF2_ITERATIONS,
    ).hex()

    return hmac.compare_digest(derived, expected)
```
## Walidacja danych użytkownika
Mechanizm walidacji w tym module jest analogiczny do tego, który był wykorzystywany w poprzednich projektach.

Rejestracja użytkownika wymaga walidacji: 
* email ma poprawny format,
* hasło spełnia wymagania bezpieczeństwa:
  * ma co najmniej 8 znaków,
  * zawiera przynajmniej jedną wielką literę,
  * zawiera przynajmniej jedną cyfrę,
  * zawiera przynajmniej jeden znak specjalny,
* hasło i jego potwierdzenie są identyczne,
* imię i nazwisko mają sensowną długość,
* konto z podanym adresem email nie istnieje już wcześniej w bazie.

Nowym elementem w tym module jest walidacja złożoności hasła.
Realizowane jest to przy użyciu wyrażeń regularnych:

```python
# app/identity/service/auth_validators.py
SPECIAL_CHARACTER_PATTERN = re.compile(r"[!@#$%^&*()_\-+=\[{\]};:'\",<.>/?\\|`~]")
UPPERCASE_PATTERN = re.compile(r"[A-Z]")
DIGIT_PATTERN = re.compile(r"\d")
```
Walidacja:
```python
# app/identity/service/auth_validators.py
def validate_password_strength(password: str):
    if len(password) < 8:
        raise ValidationError("Hasło musi mieć co najmniej 8 znaków.")

    if UPPERCASE_PATTERN.search(password) is None:
        raise ValidationError("Hasło musi zawierać co najmniej jedną wielką literę.")

    if DIGIT_PATTERN.search(password) is None:
        raise ValidationError("Hasło musi zawierać co najmniej jedną cyfrę.")

    if SPECIAL_CHARACTER_PATTERN.search(password) is None:
        raise ValidationError("Hasło musi zawierać co najmniej jeden znak specjalny.")
```

## Logowanie i utrzymywanie sesji użytkownika

W module `identity` wprowadzony został mechanizm logowania oparty o sesje przechowywane po stronie serwera oraz cookie po stronie klienta.

Backend przechowuje pełną informację o sesji użytkownika w bazie danych, natomiast przeglądarka zapisuje jedynie identyfikator tej sesji w postaci cookie (`auth_token`). Przy każdym kolejnym żądaniu cookie to jest automatycznie dołączane przez przeglądarkę, co pozwala backendowi rozpoznać użytkownika bez konieczności ponownego logowania.

### Proces logowania
Po wywołaniu endpointu `POST /auth/login` system wykonuje następujące kroki:
1. wyszukuje użytkownika po adresie email,
2. weryfikuje poprawność hasła,
3. sprawdza, czy konto jest aktywne (w systemach produkcyjnych konto najczęściej aktywowane jest dopiero po potwierdzeniu adresu email),
4. generuje nowy token sesji,
5. zapisuje sesję w bazie danych (`operator_sessions`),
6. ustawia cookie w odpowiedzi HTTP.

    <img src="images/Lab_7/cookie.png" width="70%">
</p>

Fragment odpowiedzialny za logowanie wraz z ustawieniem cookie:
```python
# app/identity/web/routes.py
@router.post("/auth/login", response_model=LoginResponse)
def login_endpoint(
    payload: LoginRequest,
    response: Response,
    auth_token: str | None = Cookie(default=None), 
    db: Session = Depends(get_db),
):
    try:
        if auth_token is not None:
            logout_operator(db, auth_token)

        operator, session_token = login_operator(
            db=db,
            email=payload.email,
            password=payload.password,
        )

        response.set_cookie(
            key="auth_token",
            value=session_token,
            httponly=True,
            samesite="lax",
            secure=False,
            max_age=SESSION_MAX_AGE_SECONDS,
        )

        return LoginResponse(operator=operator)

    except AuthenticationError as e:
        raise HTTPException(status_code=401, detail=str(e))

    except AuthorizationError as e:
        raise HTTPException(status_code=403, detail=str(e))
```

Znaczenie parametrów cookie:
* auth_token – identyfikator sesji użytkownika,
* httponly=True – cookie nie jest dostępne z poziomu JavaScript (zwiększa bezpieczeństwo),
* samesite="lax" – ogranicza przesyłanie cookie w kontekście cross-site,
* secure=False – cookie działa po HTTP (w produkcji powinno być True),
* max_age= 1800 – czas życia cookie w sekundach (30 minut).

### Sesja
Sesja użytkownika składa się z dwóch elementów:
* identyfikatora przechowywanego po stronie klienta (cookie),
* rekordu sesji przechowywanego w bazie danych.

Model ORM sesji po stronie serwera:

### Podtrzymywanie sesji
W systemie zastosowano mechanizm **sliding session**, czyli sesji przesuwanej w czasie.

Oznacza to, że sesja nie wygasa po stałych 30 minutach od momentu logowania, lecz po 30 minutach braku aktywności użytkownika.

Aby mechanizm działał poprawnie, kontrola sesji odbywa się zarówno po stronie klienta, jak i po stronie serwera. Po stronie klienta odświeżany jest czas życia cookie (`max_age`), natomiast po stronie serwera aktualizowany jest czas ostatniej aktywności sesji (`last_used_at`).

#### Po stronie serwera
W warstwie service zdefiniowana jest maksymalna długość życia sesji:
```python
# app/identity/service/auth_service.py
SESSION_MAX_AGE_SECONDS = 1800
```
Następnie system sprawdza, czy od ostatniego użycia sesji nie minęło zbyt dużo czasu:
```python
# app/identity/service/auth_service.py
def is_session_expired(last_used_at: datetime) -> bool:
    return datetime.now() - last_used_at > timedelta(seconds=SESSION_MAX_AGE_SECONDS)
```
Jeżeli sesja wygasła, zostaje ona usunięta z bazy danych, a użytkownik dostaje informację, że sesja wygasła.
```python
# app/identity/service/auth_service.py
def get_current_operator(db: Session, session_token: str) -> OperatorORM:
    session = get_session_by_token(db, session_token)
    if session is None:
        raise AuthorizationError("Brak aktywnej sesji.")

    if is_session_expired(session.last_used_at):
        delete_session_by_id(db, session.id)
        raise AuthorizationError("Sesja wygasła.")

    operator = get_operator_by_id(db, session.operator_id)
    if operator is None:
        delete_session_by_id(db, session.id)
        raise AuthorizationError("Operator przypisany do sesji nie istnieje.")

    if not operator.is_active:
        delete_session_by_id(db, session.id)
        raise AuthorizationError("Konto operatora jest nieaktywne.")

    update_session_last_used(db, session)
    return operator
```
System nie tylko odrzuca starą sesję, ale również aktywnie usuwa ją z bazy. Dzięki temu nie gromadzą się nieużywane rekordy sesji.

#### Po stronie klienta
Samo zaktualizowanie `last_used_at` w bazie nie wystarczy, ponieważ cookie po stronie klienta również ma swój czas życia. Dlatego przy poprawnym użyciu endpointu GET /auth/me cookie jest ustawiane ponownie.

Endpoint `/auth/me` służy do pobrania danych aktualnie zalogowanego użytkownika.

```python
# app/identity/web/routes.py
@router.get("/auth/me", response_model=OperatorResponse, status_code=status.HTTP_200_OK)
def me_endpoint(
    response: Response,
    auth_token: str | None = Cookie(default=None),
    db: Session = Depends(get_db),
):
    if auth_token is None:
        raise HTTPException(status_code=401, detail="Brak aktywnej sesji.")

    try:
        operator = get_current_operator(db, auth_token)

        response.set_cookie(
            key="auth_token",
            value=auth_token,
            httponly=True,
            samesite="lax",
            secure=False,
            max_age=SESSION_MAX_AGE_SECONDS,
        )

        return operator

    except AuthorizationError as e:
        raise HTTPException(status_code=401, detail=str(e))
```


Parametr:

```python
auth_token: str | None = Cookie(default=None)
```
oznacza, że FastAPI automatycznie odczytuje wartość cookie z żądania HTTP i przekazuje ją do funkcji endpointu. Na tej podstawie backend może wyszukać sesję w bazie danych i ustalić, który operator jest aktualnie zalogowany.

### Wylogowanie użytkownika
Wylogowanie polega na równoczesnym usunięciu sesji po stronie serwera jak i po stronie klienta.

```python
# app/identity/web/routes.py
@router.post("/auth/logout", status_code=status.HTTP_204_NO_CONTENT)
def logout_endpoint(
    response: Response,
    auth_token: str | None = Cookie(default=None),
    db: Session = Depends(get_db),
):
    if auth_token is not None:
        logout_operator(db, auth_token)

    response.delete_cookie(
        key="auth_token",
        httponly=True,
        samesite="lax",
        secure=False,
    )
    response.status_code = 204
    return response
```

### Przelogowanie użytkownika
Jeżeli użytkownik zaloguje się ponownie, np. na inne konto, system usuwa poprzednią sesję z bazy danych, a następnie tworzy nową. W danym momencie w przeglądarce istnieć może tylko jedna aktywna sesja, a baza danych nie przechowuje nieużywanych rekordów sesji. Każde nowe logowanie zastępuje poprzednie.

```python
if auth_token is not None:
    logout_operator(db, auth_token)
```
