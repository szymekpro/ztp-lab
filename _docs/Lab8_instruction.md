# Laboratorium 8

Celem Laboratorium 8 jest wprowadzenie systemu koszyka. W poprzednich laboratoriach system opierał się głównie na operacjach CRUD, czyli bezpośrednich operacjach na pojedynczych rekordach. Każde zapytanie wykonywało jedną prostą operację na bazie danych. W tym laboratorium wprowadzamy nowy sposób działania, w którym jedno żądanie użytkownika może inicjować wykonanie wielu operacji po stronie systemu, a ich rezultatem jest jeden spójny efekt biznesowy.

System zostaje rozszerzony o moduł `group_assignment`, który umożliwia budowanie roboczej grupy studentów oraz jej zatwierdzanie jako jednej operacji.
Moduł `group_assignment` pełni w tym laboratorium rolę analogiczną do koszyka w systemach e-commerce. Studenci odpowiadają produktom, a proces przypisania odpowiada procesowi checkout.

## Ścieżka użytkownika po aplikacji po Laboratorium 8
Po wprowadzeniu zmian użytkownik(operator) przechodzi przez system w sposób sekwencyjny.Najpierw rejestruje konto i loguje się do aplikacji. W wyniku logowania tworzona jest sesja zapisana w ciasteczku auth_token, które jest automatycznie przesyłane przy kolejnych zapytaniach.

Następnie użytkownik pobiera swoją roboczą grupę studentów (draft). Jeśli jeszcze nie istnieje, system tworzy ją automatycznie. Operator może dodawać i usuwać studentów z tej grupy, a system na bieżąco przelicza liczbę studentów oraz sumę punktów ECTS.

Po przygotowaniu grupy użytkownik zatwierdza przypisanie. System tworzy trwały rekord przypisania, zapisuje studentów i generuje numer w formacie `PRZ-YYYYMMDD-000001`. Po tej operacji robocza grupa zostaje wyczyszczona.

Operator może również przeglądać historię swoich przypisań oraz szczegóły pojedynczego przypisania.

## Moduł group_assignment
Moduł ten powstał ze względu na to, że operacja przypisania nie jest jedną operacją na jednym zasobie, lecz szeregiem operacji obejmującym wiele kroków. Wprowadzenie osobnego modułu pozwala wydzielić tę logikę i nie mieszać jej z istniejącymi modułami, takimi jak `students` czy `identity`.

Do odpowiedzialności tego modułu należą:
* zarządzanie roboczą grupą studentów (koszykiem)
* dodawanie i usuwanie studentów z grupy
* realizacja przypisania (zakup w koszyku)
* przechowanie historii przypisań studentów (historii zamówień)


## Model danych w bazie danych
Do obsługi procesu przypisywania studentów do grup dodane zostały cztery nowe tabele.

Tabela `assignment_drafts` przechowuje roboczą grupę operatora(koszyk). Jest to miejsce, w którym operator może gromadzić studentów przed finalnym zatwierdzeniem operacji. Każdy draft jest przypisany do konkretnego operatora przez pole `operator_id`.

```sql
-- app/db/init/004_group_assignment.sql
CREATE TABLE IF NOT EXISTS assignment_drafts (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    operator_id INTEGER NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```
Dzięki temu system wie, która robocza grupa należy do aktualnie zalogowanego operatora.


Tabela `assignment_draft_items` przechowuje pozycje roboczej grupy, czyli studentów dodanych do draftu (koszyka). 

```sql
-- app/db/init/004_group_assignment.sql
CREATE TABLE IF NOT EXISTS assignment_draft_items (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    draft_id INTEGER NOT NULL REFERENCES assignment_drafts(id) ON DELETE CASCADE,
    student_id INTEGER NOT NULL REFERENCES students(id) ON DELETE CASCADE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uq_assignment_draft_student UNIQUE (draft_id, student_id)
);
```
Pole `student_id` wskazuje konkretnego studenta, natomiast `draft_id` wskazuje roboczą grupę, do której został dodany. Ograniczenie UNIQUE (`draft_id`, `student_id`) zabezpiecza system przed dodaniem tego samego studenta do tej samej grupy więcej niż raz.


Tabela `assignment_batches` przechowuje zatwierdzone przypisania (wykonane zamówienia z koszyka). W przeciwieństwie do draftu jest to już trwały rekord operacji, który zostaje w historii systemu na zawsze.

```sql
-- app/db/init/004_group_assignment.sql
CREATE TABLE IF NOT EXISTS assignment_batches (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    operator_id INTEGER NOT NULL REFERENCES operators(id) ON DELETE CASCADE,
    assignment_number VARCHAR NOT NULL UNIQUE,
    status VARCHAR NOT NULL DEFAULT 'PENDING',
    students_count INTEGER NOT NULL DEFAULT 0,
    total_ects INTEGER NOT NULL DEFAULT 0,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```
Tabela przechowuje dane podsumowujące całą zatwierdzoną operację. Należą do nich numer przypisania, status, liczbę studentów oraz sumę punktów ECTS.

Tabela `assignment_batch_items` przechowuje studentów zapisanych w zatwierdzonym przypisaniu (przechowuje szczegóły zamówienia).

```sql
-- app/db/init/004_group_assignment.sql
CREATE TABLE IF NOT EXISTS assignment_batch_items (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    batch_id INTEGER NOT NULL REFERENCES assignment_batches(id) ON DELETE CASCADE,
    student_id INTEGER NOT NULL REFERENCES students(id) ON DELETE RESTRICT,
    student_name VARCHAR NOT NULL,
    student_lastname VARCHAR NOT NULL,
    student_code VARCHAR NOT NULL,
    ects_points INTEGER NOT NULL
);
```

W tej tabeli oprócz `student_id` zapisujemy również dane studenta, takie jak imię, nazwisko, numer albumu i punkty ECTS. Jest to *snapshot danych*, czyli zapis stanu studenta z momentu zatwierdzenia przypisania. Dzięki niemu późniejsza edycja danych studenta nie zmienia historii wcześniejszych przypisań.

## Modele, schematy i repozytorium 
Po utworzeniu tabel w bazie danych, do projektu dodane zostały modele ORM, schematy Pydantic, repozytorium. 

### Modele ORM
Najważniejsze zadeklarowane relacje to
draft → items, 
batch → items, 
draft item → student

Oznacza to, że z poziomu roboczej grupy można pobrać jej pozycje, z poziomu zatwierdzonego przypisania można pobrać zapisanych studentów, a każda pozycja draftu może odwołać się do konkretnego studenta.

Najważniejszy fragment modelu dotyczy relacji draftu z jego pozycjami:
```python
# app/group_assignment/model/assignment_draft_orm.py
items = relationship(
    "AssignmentDraftItemORM",
    cascade="all, delete-orphan",
    lazy="selectin",
)
```
Ta relacja pozwala na traktowanie draftu(koszyka) jako obiekt posiadający listę pozycji. Ustawienie cascade="all, delete-orphan" powoduje, że pozycje draftu są zależne od draftu i mogą być usuwane razem z nim (tak jak produkt wykupiony ze sklepu powinien zniknąć z półki).

### Schematy Pydantic
`AssignmentDraftItemCreate` jest modelem wejściowym używanym przy dodawaniu studenta do grupy. Klient przekazuje tylko `student_id`, ponieważ operator nie jest podawany w body requestu, lecz wynika z aktywnej sesji.

`AssignmentDraftResponse` zwraca aktualny stan draftu, czyli listę studentów, liczbę pozycji oraz sumę punktów ECTS(analogia do ceny produktów w koszyku). 

`AssignmentBatchResponse` zwraca szczegóły zatwierdzonego przypisania wraz z numerem, statusem i zapisanymi studentami. 

`AssignmentBatchListItemResponse` jest uproszczoną wersją odpowiedzi używaną przy pobieraniu listy przypisań, aby nie zwracać pełnych szczegółów każdego rekordu.

### Repozytorium
Najważniejszą funkcją repozytorium jest `get_or_create_draft`, która automatycznie tworzy roboczą grupę, jeśli jeszcze nie istnieje. Dzięki temu nie jest potrzebny osobny endpoint do jej zakładania.

```python
# app/group_assignment/data/assignment_repository.py
def get_or_create_draft(
    db: Session,
    operator_id: int,
) -> AssignmentDraftORM:
    draft = get_draft_by_operator_id(db, operator_id)

    if draft is not None:
        return draft

    return create_draft(db, operator_id)
```

## Warstwa service
Najważniejszym nowym elementem warstwy service jest agregacja danych. System na bieżąco oblicza liczbę studentów oraz sumę punktów ECTS.
```python
# app/group_assignment/service/assignment_service.py
def _calculate_total_ects(draft: AssignmentDraftORM) -> int:
    return sum(item.student.ects_points for item in draft.items)

def _build_draft_response(draft: AssignmentDraftORM) -> AssignmentDraftResponse:
    return AssignmentDraftResponse(
        id=draft.id,
        operator_id=draft.operator_id,
        items=[...],
        students_count=len(draft.items),
        total_ects=_calculate_total_ects(draft),
        created_at=draft.created_at,
        updated_at=draft.updated_at,
    )
```
Dzięki temu dane takie jak liczba studentów i suma ECTS są zawsze spójne z aktualnym stanem roboczej grupy i nie wymagają ręcznego zapisu w bazie.

## Proces zatwierdzania przypisania (koszyka)
Zatwierdzenie przypisania jest najważniejszą operacją w tym module. W przeciwieństwie do wcześniejszych endpointów nie jest to pojedyncza operacja CRUD, lecz proces biznesowy, który składa się z kilku powiązanych kroków wykonywanych w określonej kolejności.

Operacja ta odpowiada za przejście ze stanu roboczego (draft/koszyk) do stanu trwałego (przypisanie/zamówienie).

Cały proces realizowany jest w funkcji:

```python
# app/group_assignment/service/confirm_assignment_handler.py
def handle_confirm_assignment(
    db: Session,
    command: ConfirmAssignmentCommand,
) -> AssignmentBatchResponse:
```

Funkcja ta nie zawiera całej logiki w jednym miejscu, lecz odwołuje się do mniejszych funkcji pomocniczych. Dzięki temu kod odzwierciedla kolejne etapy procesu.

### Pobranie i walidacja roboczej grupy
Na początku system pobiera draft operatora i sprawdza, czy zawiera on jakiekolwiek dane:

```python
def _get_valid_draft(...):
    draft = get_or_create_draft(db, operator_id)

    if not draft.items:
        raise AssignmentValidationError("Nie można zatwierdzić pustej grupy.")
```
Na tym poziomie odbywa się walidacja całego procesu, ponieważ nie sprawdzamy pojedynczego pola, lecz stan całej operacji.

### Agregacja danych
System oblicza dane podsumowujące:
```python
def _calculate_total_ects(draft):
    return sum(item.student.ects_points for item in draft.items)
```
### Utworzenie rekordu przypisania/zamówienia

```python
# app/group_assignment/service/confirm_assignment_handler.py

def _create_assignment_batch(
    db: Session,
    operator_id: int,
    students_count: int,
    total_ects: int,
) -> AssignmentBatchORM:
    batch = AssignmentBatchORM(
        operator_id=operator_id,
        assignment_number="TEMP",
        status="PENDING",
        students_count=students_count,
        total_ects=total_ects,
    )

    batch = add_assignment_batch(db, batch)

    batch.assignment_number = _generate_assignment_number(batch.id)

    db.add(batch)
    db.commit()
    db.refresh(batch)

    return batch
```

Po zapisie generowany jest numer przypisania(zamówienia):
```python
# app/group_assignment/service/confirm_assignment_handler.py
def _generate_assignment_number(batch_id: int) -> str:
    today = datetime.now().strftime("%Y%m%d")
    return f"PRZ-{today}-{batch_id:06d}"
```
Ma on postać: `PRZ-YYYYMMDD-000001`.

### Utworzenie pozycji przypisania
Po utworzeniu batcha system przepisuje studentów z draftu
```python
_create_assignment_batch_items(
    db=db,
    batch_id=batch.id,
    draft=draft,
)
```

Jest to tzw. snapshot danych. Zapisywany jest stan studenta w momencie zatwierdzania przypisania. Dzięki temu w przypadku gdyby liczba punktów ECTS studenta uległa zmianie, nie wpłynie ona na wynik przypisania. Analogicznie w przypadku zamówienia, jeśli cena jednego z zamówionych produktów uległaby zmianie, cena zamówienia, jak i produktu w zamówieniu nie zmieni się. 

### Czyszczenie roboczej grupy
Po zakończeniu procesu robocza grupa zostaje wyczyszczona, a system zwraca dane zatwierdzonego przypisania.
```python
    clear_draft_items(db, draft.id)
    return _build_batch_response(batch)
```
Odpowiedź zawiera numer przypisania, status, liczbę studentów, sumę punktów ECTS oraz listę studentów.

Jest to przykład operacji, w której jedno żądanie HTTP uruchamia wiele powiązanych operacji na bazie danych i prowadzi do powstania jednego spójnego efektu biznesowego.

## Command Pattern
Command Pattern to wzorzec projektowy, którego głównym celem jest oddzielenie informacji o tym co chcemy zrobić od kodu, który to wykonuje. W praktyce oznacza to, że zamiast przekazywać dane operacji bezpośrednio do funkcji, tworzymy obiekt opisujący tę operację, a następnie przekazujemy go do miejsca, które ją realizuje.

Takie podejście zaczyna mieć sens w momencie, gdy operacje w systemie przestają być proste. W poprzednich laboratoriach większość działań miała charakter CRUD, np. utworzeniu studenta lub jego edycji. W takim przypadku nie było potrzeby dodatkowego rozdzielania logiki, ponieważ operacja była krótka i jednoznaczna.

Sytuacja zmienia się wtedy, gdy pojawia się proces biznesowy. Zatwierdzenie przypisania studentów nie polega na zapisaniu jednego rekordu, lecz na wykonaniu całej sekwencji działań: pobraniu roboczej grupy, walidacji grupy, obliczeniu danych zagregowanych, utworzeniu przypisania, zapisaniu studentów oraz wyczyszczeniu draftu. Umieszczenie całej tej logiki bezpośrednio w endpointcie lub w jednej funkcji z wieloma parametrami szybko prowadzi do utraty czytelności kodu.

W tym miejscu pojawia się Command Pattern. Zamiast wywoływać funkcję w stylu:
```python
confirm_assignment(operator_id)
```
tworzymy obiekt reprezentujący zamiar wykonania operacji:
```python
command = ConfirmAssignmentCommand(operator_id)
```
a następnie przekazujemy go do handlera:
```python
handle_confirm_assignment(command)
```

Command nie zawiera logiki, a jego rolą jest przechowywanie danych potrzebnych do wykonania operacji. W naszym przypadku jest to `operator_id`, na podstawie którego system odnajduje roboczą grupę użytkownika. Cała logika znajduje się w handlerze, który realizuje kolejne kroki procesu.

Dzięki temu kod jest czytelniejszy- funkcja `handle_confirm_assignment` opisuje przebieg operacji, a endpoint ogranicza się do obsługi HTTP i przekazania żądania dalej.

W naszym przypadku Command Pattern ma uproszczoną formę i technicznie operacja mogłaby zostać wykonana zwykłą funkcją. Jego zastosowanie pozwala jednak traktować zatwierdzenie jako osobną operację biznesową, którą opisujemy jednym obiektem i przekazujemy do handlera.

Podejście to nabiera znaczenia wraz ze wzrostem złożoności operacji, np. przy dodawaniu powiadomień, różnych trybów wykonania lub dodatkowych parametrów.
